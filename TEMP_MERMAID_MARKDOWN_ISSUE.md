# Mermaid Rendering Hardening and Markdown-First UI Contract

## Summary

Fred should stay aligned with standard chat ecosystems and keep its message protocol simple, predictable, and easy to evolve. Today, Mermaid and code are carried as markdown text and rendered by the frontend, while structured `ui_parts` are used for explicit widgets like links and geo content. That split is correct, but it must be documented clearly and hardened in the UI so future work does not drift into a bespoke protocol for every rich content type.

This issue intentionally combines the design decision and implementation execution in one place.

## Problem

Mermaid diagrams generated by the model can fail to render because Mermaid syntax is strict and LLM output can be slightly malformed. In parallel, streaming markdown can temporarily contain incomplete fenced blocks, which causes avoidable rendering noise or parse errors.

We need a robust, minimal solution that preserves a standard markdown-first model.

## Decision (Design Contract)

Keep the protocol standard and markdown-first:

- Prose, code fences, math, and Mermaid remain plain markdown text.
- `ui_parts` remain reserved for explicit, typed widgets that the frontend can render without parsing free text.
- Do not introduce structured `code` or `diagram` parts unless a concrete UI need proves markdown is insufficient.
- Backend emits semantic events and typed widgets only where deterministic rendering is required.
- Frontend owns markdown rendering behavior, streaming guards, and Mermaid parsing resilience.

This keeps Fred close to OpenWebUI / OpenAI-style message ecosystems while still enabling explicit structured widgets when warranted.

## Scope

### In Scope

- Document the markdown-first vs structured-parts rule in the runtime design contract.
- Harden Mermaid rendering with fallback sanitization for common LLM syntax issues.
- Add streaming guard behavior so incomplete fenced blocks are not rendered mid-stream.
- Stabilize markdown renderer component mapping to reduce unnecessary remount/re-render churn.

### Out of Scope

- Introducing new protocol primitives for `code`/`diagram` in runtime events.
- Changing runtime to emit Mermaid as structured content.
- Redesigning the chat protocol into a custom rich-content transport.

## Implementation Checklist

- [x] Add representation rule to runtime design contract.
- [x] Add markdown streaming guard for unclosed fenced blocks.
- [x] Add Mermaid sanitizer fallback (raw render first, sanitize only on failure).
- [x] Keep valid Mermaid unchanged whenever raw parsing succeeds.
- [x] Forward streaming state into markdown renderer.
- [x] Stabilize markdown renderer components reference with memoization.
- [x] Add tests for streaming guard behavior.

## Acceptance Criteria

- Valid Mermaid diagrams render without sanitizer mutation.
- Common malformed Mermaid output from LLMs can recover through fallback sanitization.
- Streaming partial markdown does not trigger avoidable parser/render failures.
- Design docs explicitly state markdown-first representation for prose/code/math/Mermaid.
- Design docs explicitly reserve `ui_parts` for typed UI widgets.
- The approach remains ecosystem-standard and avoids protocol over-specialization.

## Why This Matters

This is a foundation guardrail. It keeps Fred predictable for SDK and agent work, avoids repeated protocol debates, and reduces future maintenance cost by aligning with well-known chat/message conventions.
