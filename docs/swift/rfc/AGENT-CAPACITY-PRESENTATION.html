Agent Capacities

One abstraction for modular agent features — tools, UI, state and policy as one vertical module instead of code scattered across the platform.

Florian Muller · July 2026
Full proposal: docs/swift/rfc/AGENT-CAPACITY-RFC.md

The premise

Nobody will write agent code.

  • The ReAct loop is a solved problem — reason, call tools, repeat. It already runs in production in fred-runtime. We will not out-engineer it per feature.
  • Building a new agent is now context engineering, not software engineering.
  • So the platform's job is to make each of these three things cheap to author and share.

A new agent = a new context

1 · System prompt
who the agent is, how it behaves
2 · Skills
reusable know-how, prompt fragments
3 · Well-integrated tools
capabilities the agent can act with

Prompts and skills are text — easy. This talk is about №3.

The premise, part two

"Well-integrated" is the hard part

Shipped on main these last weeks: PPT filler · WritableDocument · document tools (RAG). None of them was "just write a tool function." Each one needed:

  • Custom chat UI — a preview card, a download chip, a side panel
  • Backend endpoints — validate an upload, analyze a template, export a file
  • Conversation-state edits — hidden system messages so the agent knows the user edited something
  • Custom tool parameters — runtime context the LLM must never see in the tool signature
  • Persisted state — its own tables, its own stored assets
  • Agent-creation & chat-time options — config forms, composer controls

Every one of those pieces landed in a different central file. Three features in, the pattern was unmistakable.

The proposal

One abstraction: the capacity

Everything an agent is given beyond its base prompt is a capacity.
A capacity = a runtime middleware + a manifest. There is no second concept.

  • MCP is not special. A remote MCP server is just McpCapacity parameterized by a server config. Built-in tools, PPT filler, WritableDocument, document-access — all capacities.
  • A capacity owns its whole vertical: tools, config fields, chat controls, chat cards, side panels, routes, tables, per-team policy — declared together, registered once.
  • One registry · one product contract · one Tools tab in the product.
Anatomy · 1/2

The manifest — declare the whole vertical

class CapacityManifest(BaseModel):
    id: str                                  # "ppt_filler", "document_access", "mcp:<server>"
    version: str                             # bumped per release — cache key for computed surfaces
    name: str; description: str; icon: str

    config_fields: list[ManagedAgentFieldSpec]  # agent-creation form — existing renderer, zero new UI
    assets: list[AssetSlot]                  # named upload slots — accepted types + min/max count each
    chat_parts: list[type[UiPart]]           # custom inline chat cards → extend the typed union
    side_panels: list[SidePanelSpec]         # panels mounted beside the chat

    router: APIRouter | None                 # auto-mounted under /capacities/{id}/...
    tables: list[type[DeclarativeBase]]      # capacity-owned; migrations ship with the package
    required_env: list[str]                 # env vars this capacity needs — checked at pod boot

    team_scope: TeamScopePolicy              # default_on | admin_gated (ReBAC, Tier 3)

Registration reads the manifest and does the ~26-file dance once, centrally, for every capacity — including extending the OpenAPI-generated part union.

Anatomy · 2/2

The capacity class — behavior + typed models

class AgentCapacity(ABC, Generic[ConfigT, StoredT, TurnOptionsT]):
    manifest: ClassVar[CapacityManifest]

    ConfigModel:       type[BaseModel]   # what the agent creator fills in (drives config_fields)
    StoredConfigModel: type[BaseModel]   # what the platform persists; = ConfigModel by default
    TurnOptionsModel:  type[BaseModel]   # what the chatting user sends each turn; EmptyModel if none
    TeamSettingsModel: type[BaseModel]   # what the team admin sets at enablement (Tier 3)

    async def validate_config(self, config: ConfigT, uploads: Mapping[str, list[UploadedFile]], ctx: SaveContext) -> StoredT:
        """Save time: parse the uploads (keyed by slot), raise typed 422s, store the assets, enrich."""
    def chat_controls(self, config: StoredT) -> list[ChatControlSpec]:
        """Chat-time surface, computed per instance — a function, not a visible_when language."""
    def upgrade_config(self, stored: dict, from_version: str) -> StoredT:
        """Optional: migrate an old stored shape forward when the manifest version bumps."""
    def middleware(self, ctx: CapacityContext[StoredT, TurnOptionsT]) -> AgentMiddleware:
        """The runtime half: tools, hooks, state — stock LangChain middleware."""
  • Every model is exported to OpenAPI → the frontend gets generated types (PptFillerTurnOptions, …) end to end.
  • Tools receive only LLM args — identity, config and services arrive through the middleware closure.
Anatomy · who configures what, when

Four layers of typed narrowing

enables it for the team, with settings — "our drive is rooted at folder 123"
TeamSettingsModel
Agent creator
configures the instance — scope, template upload, defaults

Chatting user

Each layer narrows the previous one, each is a typed Pydantic model, and none of them ever appears in a tool signature.

Runtime

The runtime half is stock LangChain middleware

Not a Fred-invented hook system. AgentMiddleware already runs in production for Deep agents. Every capacity need maps to an existing hook:

Capacity needLangChain primitive
Add tools to the agentmiddleware.tools
Build a tool dynamically per turn (PPT filler: schema from the parsed template)wrap_model_call → edits request.tools
Edit conversation state (inject a hidden system note)before_model + reducers
Custom conversation state (the draft document itself)state_schema
Runtime/LLM split (identity, config, services)context_schema → runtime.context

Free with the ecosystem: retries, model fallback, tool-call limits, summarization, HITL — and Fred capacities become publishable LangChain middleware.

Frontend

One plugin per capacity, four registries

// rework/features/capacities/ppt_filler/index.ts
export const pptFillerCapacity: CapacityUiPlugin = {
  id: "ppt_filler",
  configWidgets:    { template_upload: TemplateUploadField },
  chatTurnControls: { /* none */ },
  partRenderers:    { ppt_preview: PptPreviewCard },
  sidePanels:       { ppt_preview_pane: PptPreviewPane },
};

One folder per capacity, registered in one index — the only shared edit.

  • Part-renderer registry — dispatch by part type; unknown parts are kept and skipped, never dropped
  • Composer control slot — renders the chat_controls() descriptors; stock kit for simple rows
  • Side-panel slot — a reserved right column capacities mount into
  • Form-widget registry — complex agent-creation fields; scalars stay fully generated

Chat cards stay strongly typed end to end: capacity parts extend the UiPart union → OpenAPI regen → typed renderers. No generic JSON envelope.

PPT filler shipped on mainRFC on main

The experience

  • Creator uploads a .pptx template with placeholders — bad templates rejected with precise errors at save time
  • Agent fills the deck from the conversation; a preview card appears in the chat
  • A resizable PDF preview pane opens beside the conversation

Where it lives as a capacity

  • dynamic tools — the fill tool's schema is generated per turn from the parsed template
  • chat_parts + side_panels — preview card + PDF pane
  • router — the pre-save analyze endpoint

Today: ~26 files across three apps. As a capacity: one package + one plugin folder.

The experience

  • User and agent co-write a document — the agent drafts, the user edits directly in an editor panel

Example 3/5

Jira ticket writer what we want next

toolsdynamic toolsconfig fieldsasset + validatechat controlschat partsside panelroutestablesstate editsHITLteam settings

The experience

  • Agent proposes a ticket as a rich preview card — title, description, labels, estimate
  • User edits the card directly, or asks the agent to revise it — direct edits notify the agent via system note
  • When ready, user clicks Create — the platform files it in Jira

Where it lives as a capacity

  • chat_parts — the editable ticket card
  • state_schema — the draft lives in agent state; before_model injects the edit notice
  • HITL — "Create" is a human-approved action, not an autonomous tool call
  • team_settings — Jira project + credentials, per team

This is WritableDocument + HITL, recombined. Zero new platform code — that's the abstraction working.

Example 4/5

AskUserQuestion what we want next

toolsdynamic toolsconfig fieldsasset + validatechat controlschat partsside panelroutestablesstate editsHITLteam settings

The experience

  • Mid-task, the agent asks a question with ready-made answer buttons — "Which environment? [dev] [staging] [prod]"
  • The turn pauses; the user clicks; the answer flows back and the agent continues
  • The same pattern Claude Code and other agentic tools use — structured clarification instead of prose ping-pong

Where it lives as a capacity

  • tools — one ask_user_question tool with typed options
  • chat_parts — the clickable question card
  • HITL interrupt — the pause/resume is the stock interrupt mechanism
  • state — the recorded answer joins the conversation state

Four slots, no config, no panels, no routes — a tiny capacity. The abstraction scales down as well as up.

Example 5/5 · a thought experiment, not a plan

Knowledge Flow as a capacity far horizon

The whole KF surface, mapped

  • vector search & doc tools → capacity tools (already happening — #1906)
  • ingestion, processors, corpus admin → router + tables
  • library pickers, scope narrowing → config_fields + chat_controls
  • per-team stores & embedder → team_settings, to the letter
  • management UI → a full page — the one slot we don't have

Why show this

Not to migrate KF — it works, and the move would be enormous. It's a design test: if the capacity abstraction could not in principle absorb our largest subsystem, it would be too small.

Practical consequence: a future page-slot must be a natural extension of the side-panel slot — not a different mechanism.

Open question · the word itself

What do we call it — for users?

The word is user-facing: it replaces the vague "tool" when a non-technical creator enables things on their agent. Priority = instantly clear in English and French, one coherent word across UI and code, no clash with an existing agentic term.

EnglishFrenchCognate?User reads it asWatch-out
Capability Capacité near "what my agent is capable of" matches the RFC prose; FR term is more natural than EN — recommended
Module Module exact "a building block I add" very clear, zero clash; a touch technical / less "what it can do"
Talent Talent exact "something my agent is good at" warm & intuitive; may read playful for enterprise admins
Extension Extension exact "an add-on for my agent" perfect cognate; implies bolt-on, not the composition model
Aptitude Aptitude exact "a knack for X" clear, slightly formal; less common as a code identifier
Capacity (current) Capacité EN drifts "capacity for what?" EN collides with scaling / capacity-planning; ambiguous to laypeople
