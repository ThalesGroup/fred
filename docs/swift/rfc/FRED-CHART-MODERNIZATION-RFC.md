## 1. Why This RFC Exists

`deploy/charts/fred` still describes the legacy runtime topology centered on
`agentic-backend`, while the platform architecture has already moved to:

- `apps/fred-agents` for runtime execution
- `apps/control-plane-backend` for tenancy, enrollment, and execution grants
- `knowledge-flow-backend` for retrieval and ingestion
- `frontend` as the browser entrypoint

This drift now creates three operational problems:

1. the base chart deploys the wrong runtime service and wrong runtime base path
2. local Helm overlays and CI still assume legacy image names and legacy chart values
3. the chart still carries startup/config assumptions that no longer match the
   first-party `fred-agents` pod

This RFC defines the monorepo chart migration path. It does **not** replace the
generic runtime discovery rules already defined in `AGENTIC-POD-RFC.md`.

---

## 2. Scope

This RFC covers only the repository-owned deployment assets for the first-party
Fred stack:

- `deploy/charts/fred`
- `deploy/local/k3d/*.yaml`
- CI jobs that build, validate, publish, or package the corresponding images/chart
- first-party Docker packaging assumptions needed by that chart

This RFC does **not** define:

- third-party runtime pod discovery contracts
- a new generic Helm library for external agent pods
- Kubernetes auto-discovery beyond the already-existing runtime catalog contract

---

## 3. Decision

### 3.1 Keep One Platform Chart, But Modernize Its Runtime Component

The chart artifact name remains `fred`.

We do **not** create a second chart such as `fred-agents`.

Instead, the existing `deploy/charts/fred` chart becomes the canonical chart for
the modern topology and replaces the legacy runtime component with
`fred-agents`.

### 3.2 The Runtime Application In Chart Values Becomes `fred-agents`

The runtime application block in Helm values becomes `applications.fred-agents`.

Kubernetes resources generated by the chart for the first-party runtime must use
the runtime name `fred-agents`, not `agentic-backend`.

This includes at minimum:

- Deployment name
- Service name
- ConfigMap/Secret names
- local Helm overlays
- image references
- ingress or reverse-proxy upstream references

### 3.3 The Canonical Runtime Base Path Is `/fred/agents/v2`

The chart must expose the first-party runtime using the same base path as the
application configuration and the control-plane runtime catalog:

`/fred/agents/v2`

The deployment target is:

- internal runtime URL:
  `http://fred-agents:8000/fred/agents/v2`
- browser-facing ingress prefix:
  `/fred/agents/v2`

The chart must avoid path rewriting whenever possible. The deployed route and
the runtime `app.base_url` must match directly.

### 3.4 The Chart Uses The Canonical Pod Entrypoint

For `apps/fred-agents`, the chart must use the canonical pod entrypoint:

`python -m fred_agents`

The chart must not keep a hand-written `uvicorn ...` command that duplicates the
startup contract already owned by `apps/fred-agents/fred_agents/__main__.py`.

Consequence:

- `app.limit_concurrency` is chart-configurable through the mounted
  `configuration.yaml`
- the pod entrypoint remains the single place that forwards this value to
  Uvicorn
- Helm does not need a second, command-line-specific source of truth for the
  same setting

### 3.5 First-Party Agent Registry Stays Code-Owned

The first-party `fred-agents` pod builds its agent registry from Python code.

Therefore the chart must **not** continue to model `agents_catalog.yaml` as a
required runtime override for that pod.

For the first-party runtime chart, the external catalog files that remain
operator-configurable are:

- `models_catalog.yaml`
- `mcp_catalog.yaml`

`conversation_policy_catalog.yaml` remains a control-plane concern.

### 3.6 Control-Plane Runtime Catalog Must Be Wired In Chart Defaults

The charted `control-plane-backend` configuration must declare the first-party
runtime explicitly under `platform.runtime_catalog_sources`.

The default runtime entry must use:

- `runtime_id: fred-agents`
- an internal `base_url` pointing to the in-cluster Service
- an ingress-safe `ingress_prefix` pointing to `/fred/agents/v2`

This is required so managed-agent enrollment and execution preparation use the
same runtime identity the chart actually deploys.

### 3.7 Local Overlays And CI Must Share The Same Naming Model

`deploy/local/k3d/*.yaml`, image build jobs, chart validation jobs, and release
packaging must all target the same modern artifact set:

- `fred-agents`
- `control-plane-backend`
- `knowledge-flow-backend`
- `frontend`

No CI or local deployment path should continue to assume that
`agentic-backend` is the runtime artifact behind `deploy/charts/fred`.

### 3.8 This Is A Breaking Chart Migration

This migration changes runtime resource names and deployment assumptions.

Operators must treat the chart change as a breaking upgrade that requires
release notes and upgrade guidance.

Per `docs/swift/platform/VERSIONING.md`, the chart version should be bumped in a
way that makes the operator review explicit before rollout.

## 4. Alternatives Considered

### 4.1 Keep `agentic-backend` As A Long-Lived Helm Alias

Rejected.

This would preserve short-term compatibility in values files, but it would also
encode a retired architecture into:

- Service names
- runtime catalog identifiers
- CI image names
- deployment documentation

That would increase migration drag and keep the chart inconsistent with
`PLATFORM_RUNTIME_MAP.md`.

### 4.2 Create A Second Chart Dedicated To `fred-agents`

Rejected.

The repository already owns one platform chart. Creating a second chart would
duplicate deployment logic while the intended topology is singular.

The correct move is to modernize the existing chart, not fork it.

### 4.3 Keep A Hand-Written Uvicorn Command In Helm

Rejected.

`apps/fred-agents` already owns the pod startup contract in
`python -m fred_agents`. Keeping a second Uvicorn command in Helm would re-open
the exact drift we are trying to remove and would make `limit_concurrency`
harder to reason about.

---

## 5. Impact On Existing OPS Tickets

### `OPS-01`

`OPS-01` becomes the chart implementation ticket for this RFC:

- rename runtime resources to `fred-agents`
- expose `/fred/agents/v2`
- wire `platform.runtime_catalog_sources`
- align local overlays with the modern topology
- carry `app.limit_concurrency` through the canonical pod entrypoint

### `OPS-02`

`OPS-02` adapts CI and release packaging so chart validation and publication use
the same modern artifact set as the chart.

### `OPS-03`

`OPS-03` aligns Dockerfiles and image packaging with the runtime the chart
deploys.

`OPS-01` remains blocked on `OPS-02` and `OPS-03`.

---

## 6. Non-Goals

- defining a generic third-party Helm distribution story for all external agent pods
- introducing Kubernetes runtime auto-discovery into the chart
- redesigning frontend transport or control-plane APIs
- adding new runtime contracts beyond what already exists in `fred-runtime`

---

## 7. Acceptance Criteria

This RFC is satisfied when all of the following are true:

- `deploy/charts/fred` deploys `fred-agents` instead of `agentic-backend`
- the deployed runtime path is `/fred/agents/v2`
- control-plane chart defaults expose `platform.runtime_catalog_sources` with
  `runtime_id: fred-agents`
- the chart uses `python -m fred_agents` as the runtime startup command
- first-party runtime chart values no longer require `agents_catalog.yaml`
- local k3d values and CI image/chart validation target the same modern naming model
