What's new
in Swift

Swift is the next generation of the Fred platform. This page walks through every significant change — from the end-user experience down to the platform administration layer.

How to read this page. Features are organized by who benefits from them most. Click Why it matters / how it works on any card to expand the design rationale, security implications, and implementation notes. The sections go from most visible (users) to least visible (platform administration).
End users New user experience
💬
Redesigned chat interface
Streaming responses via SSE, file attachments, session history, and a context picker — all in a single unified chat page with no full-page reload.
Why it matters / how it works
What changed

The previous chat view was stateless and page-reload heavy. Swift ships ManagedChatPage — a React component that holds full session state in Redux and communicates with agent pods exclusively via Server-Sent Events (SSE). Tokens stream to the screen as they are generated; there is no polling.

Why SSE, not WebSocket

SSE is unidirectional (server → client) and works transparently through HTTP/2 multiplexing, load balancers, and cloud proxies without upgrade negotiation. User messages are sent as standard POST /execute requests. This separation keeps the security model simple — a single authenticated POST initiates the stream, and the stream carries no user-initiated data.

Session continuity

Sessions are stored in the control-plane database and rehydrated on page load. The user returns to an in-progress conversation even after closing and reopening the browser, without re-authenticating the stream.

⏳
Real-time task progress
Document ingestion, agent evaluation, and configuration import all show live step-by-step progress — with a progress bar, step label, and pass/fail badge — without leaving the current page.
Why it matters / how it works
The task infrastructure

Every long-running operation (ingestion, evaluation, import) registers a task in the control plane and emits TaskEvent messages over an SSE stream at GET /tasks/{task_id}/events. The frontend subscribes immediately and updates a shared Redux store. TaskCard, TaskProgressBar, and TaskStateBadge are composable UI atoms that consume this store — any page can show progress without coupling to the specific operation.

Why a shared task store

Users often navigate away while a large document set ingests. Because task state lives in Redux (not in a single page component), the progress indicator persists across navigation. On page reload, the frontend rehydrates active tasks from all task producers (knowledge-flow, evaluation, control-plane) and reopens SSE streams from the last received sequence number — no event is lost.

Security: who sees which task

The SSE endpoint enforces authorize_task_access: access is granted to the task creator, any platform admin, or a member with read access to the task's team. A user cannot observe another user's task stream even if they guess the task ID. Canceling is stricter still — creator or platform admin only.

📚
Tag-organized knowledge base
Documents are organized in named tags (corpora). Each tag is owned by a user or a team. Agents with RAG capability search the tags they have been configured to use.
Why it matters / how it works
Three storage layers
  • Postgres (metadata table) — document identity, file info, processing stages, tag membership.
  • OpenSearch — vector embeddings for semantic search. Indexed by document_uid.
  • MinIO / Cloud Storage — raw file binaries, previews, and tabular parquet extracts. Keyed by SHA-256 hash.

The three layers are coordinated but independent. An import or mirror operation that restores only the Postgres metadata records makes the documents immediately visible to the application — as long as the corresponding binaries and vectors are also present in the stores (they may have been mirrored separately).

Tag ownership and visibility

Tag ownership is enforced by OpenFGA: a tuple user:UID owner tag:TID or team:TID owner tag:TID controls read, write, and share permissions. No application-layer ACL check is duplicated — the authorization model is the single source of truth.

🧠
Conversational memory
Agents can retain structured facts across separate sessions — decisions made in Monday's meeting are available when the user resumes on Wednesday, without re-pasting context.
Why it matters / how it works
Why persistent memory matters

LLM context windows are session-scoped. Without explicit memory, every new conversation starts from zero. For operational use-cases (recurring project reviews, multi-week analysis tasks), this breaks continuity and forces users to re-introduce background that the agent already processed.

Design: structured, not raw transcripts

Memory is stored as structured records (key-value facts, decisions, and summaries) — not as raw transcript dumps. The agent writes memory entries using a dedicated tool during the session; on the next session it retrieves the most relevant records via semantic search. This keeps injected context small and focused, avoiding token bloat.

Scope. Memory is scoped to a (user, agent) pair by default. Team-shared memory is on the roadmap.
Team admins Team configuration & management
🤖
Agent instance management
Team admins create, configure, and enable or disable agent instances from a catalog of templates — no deployment required. Each instance is a named, tunable copy of a template running in a shared agent pod.
Why it matters / how it works
Template vs. instance

A template is a versioned agent definition packaged inside an agent pod (e.g. fred-agents:fred.github.assistant). An instance is a team-scoped configuration record that points to a template and carries a display name, tuning parameters (role, description), and an enabled flag. Multiple teams can hold instances of the same template without interference.

Authorization: team boundary enforcement

Instance creation and configuration are gated by CAN_ADMIN_TEAM permission in OpenFGA. A team admin can only manage instances belonging to their own team. The control plane validates ownership on every write — there is no "edit any instance" API that relies solely on request parameters.

ExecutionGrant flow

When a user starts a chat, the control plane issues a short-lived ExecutionGrant (5 minutes, signed) that binds the user's identity, team, session, and the target agent instance. The frontend passes this grant directly to the agent pod over SSE. The pod validates the grant cryptographically without calling back to the control plane — keeping the execution path low-latency and control-plane independent.

🗂️
Team knowledge corpus
Upload documents in 31 formats (PDF, Word, CSV, code, …), organize them into named tags, and make them available to the team's RAG agents. Processing (vectorization, tabular extraction, preview) runs in the background with real-time progress.
Why it matters / how it works
Three processing profiles
  • Fast — text extraction + chunking only. Seconds per document.
  • Medium — text + vectors (OpenSearch). Standard for most use-cases.
  • Rich — text + vectors + previews + tabular extraction. Minutes per document; enables the SQL agent on structured files.
Why Temporal for ingestion

Large document sets can take minutes to process. Temporal provides durable execution: if the worker pod restarts mid-ingestion, the workflow resumes from the last checkpoint without re-processing already-completed documents. The user-visible task progress reflects this durable state.

Data isolation

Documents uploaded by a team are stored under a team-scoped key in object storage and tagged in Postgres with the team's tag IDs. A user from another team cannot search or retrieve those documents even if they know the document_uid, because the OpenFGA tag ownership tuple is absent.

📦
Configuration backup & restore
Export a team's agent instances, knowledge tags, and document metadata as a portable zip snapshot. Re-import it into another environment — or restore it after an incident — from the admin UI.
Why it matters / how it works
What the zip contains
  • manifest.json — format version, table row counts, timestamp.
  • postgres/agent.jsonl — agent instance rows.
  • postgres/tag.jsonl — tag definitions with ownership.
  • postgres/metadata.jsonl — document metadata stubs (not the file binaries).
  • openfga/tuples.json — authorization tuples (ownership, sharing).

Document binaries and vector embeddings are not in the zip — they are referenced by SHA-256 hash and must be mirrored separately between object stores. This keeps snapshots small (typically < 1 MB for most teams) and decouples configuration portability from data volume.

Atomic import

The import runs as a single database transaction: agents, tags, and document metadata are all written — or all rolled back — together. A failed import leaves the target environment completely unchanged. Re-running the same zip is safe: the importer checks for existing records before inserting (idempotent by primary key).

Agent template mapping

When importing from a kea snapshot, agent definition_ref values (e.g. v2.react.basic) are translated to their Swift equivalents (e.g. fred-agents:fred.github.assistant) via a curated mapping table. Agents with no mapping are skipped with a warning — they do not block the rest of the import.

Current scope. The import endpoint requires platform admin access. Team-scoped export (backup only your team's data) is the next step and requires scoping the OpenFGA tuple export and the Postgres queries to a single team_id.
🧪
Agent evaluation campaigns
Run reproducible quality tests against an agent instance using a predefined scenario set. Results include pass/fail per scenario, scoring rationale, and a campaign-level quality score.
Why it matters / how it works
Why evaluation matters before production

LLM-based agents are non-deterministic. A prompt change, model version bump, or RAG corpus update can silently degrade quality. Evaluation campaigns give teams a repeatable signal: run the same 20 scenarios before and after any change to detect regressions before users do.

Campaign structure

A campaign points to a scenario set (YAML files, versioned in Git) and a target agent instance. The evaluation worker executes each scenario against the live agent, then scores the response using a separate judge agent. Results are stored per-run — teams can compare campaign N with campaign N-1 to track quality trends.

No production traffic in evaluation

Evaluation runs are isolated: they use a dedicated session context and do not appear in the user-facing conversation history. The judge agent is a separate model call — it never modifies the conversation being evaluated.

Platform admins Operations & security
🔄
Platform import (kea → swift)
Upload a kea snapshot zip from the admin UI to migrate all business configuration to Swift in one atomic operation. Progress is tracked live. Interruptions leave the target clean — roll back and retry.
Why it matters / how it works
Why a service, not a pg_dump

The kea and Swift schemas diverge. Agent definition_ref values are kea-specific identifiers that do not exist in Swift — a raw SQL copy would produce broken records. The importer runs a classification pass (MAPPED / IGNORED / GAP) and translates identifiers before any database write. This logic requires code, not SQL.

Pre-conditions (handled by ops, not the importer)
  • Keycloak pre-seeded with the same user UUIDs (MIGR-04).
  • OpenFGA tuples bulk-copied from kea store to swift store (Option A).
  • MinIO object keys mirrored key-for-key (mc mirror, MIGR-06).
  • OpenSearch vector index mirrored or rebuilt.

Because document UIDs are stable across the migration, the Postgres metadata records imported by the importer immediately resolve to the mirrored binaries and vectors in the target stores — no re-ingestion required.

Observability during import

The importer emits structured MigrationTaskEvent messages over SSE as each phase completes: Classifying agents → Importing agents (N/M) → Importing tags → Importing documents. Warnings (GAP agents, missing OpenFGA tuples) appear in the final summary and in server logs. The task card stays visible until the admin acknowledges it.

🔐
Relationship-based authorization (ReBAC)
All authorization decisions are evaluated by OpenFGA against a typed authorization model. No permission is hardcoded in application logic — who can do what is fully auditable and changeable without a code deploy.
Why it matters / how it works
ReBAC vs. classic RBAC

Classic RBAC grants permissions to roles (admin can edit anything). ReBAC grants permissions based on relationships (user U is a member of team T, team T owns agent A, therefore U can use A). This is the right model for a multi-tenant platform where permissions depend on object ownership, not just the caller's role.

OpenFGA as the single source of truth

Every ownership and membership relationship — user↔team, team↔agent-instance, user↔tag, team↔tag — is stored as a tuple in OpenFGA. Application code calls rebac.check_user_team_permission_or_raise() and never re-implements the relationship graph. This means:

  • Changing the authorization model (new permission type, new relation) requires an OpenFGA model update, not a code change.
  • An external audit can enumerate all tuples to reconstruct who has access to what at any point in time.
Layered defence

API endpoints enforce both RBAC (Keycloak roles extracted from the JWT) and ReBAC (OpenFGA tuple check). A platform admin JWT does not automatically grant ownership of a team's private tag — the admin role gives access through a separate bypass path that is explicitly coded, not implicit.

📊
Observability — metrics, KPIs, tracing
Every API call writes KPI counters to ClickHouse. Prometheus metrics are exported from each service. LLM traces (token counts, latency, errors) go to Langfuse. A Grafana dashboard surfaces the full picture.
Why it matters / how it works
Three observability planes
  • Prometheus / Grafana — infrastructure health: request rate, latency P99, error rate per service.
  • ClickHouse KPIs — business events: sessions started, agents used, documents ingested, evaluation scores. Suitable for billing or usage dashboards.
  • Langfuse traces — LLM-level detail: prompt text, model, token counts, tool calls, latency per LLM call. Useful for debugging agent behaviour and cost attribution.
Why separate stores

Each plane has different query patterns and retention needs. Prometheus is optimized for real-time alerting (15-second scrape intervals, short retention). ClickHouse handles high-volume event ingestion for aggregated dashboards over weeks. Langfuse stores rich per-call detail with longer retention for debugging. Coupling them would degrade performance and increase operational complexity.

KPI middleware. A FastAPI middleware (KPIMiddleware) intercepts every HTTP response and writes a structured event to ClickHouse asynchronously — application code never calls a metrics SDK directly.
