# Session Identity Contract

## 1 Purpose

This document is the single source of truth for how conversations are identified,
stored, and made visible to administrators in the Fred platform.

It answers three questions that every developer and operator must be able to
answer without reading code:

1. What is a session and what identifies it?
2. Where does conversation data live and who owns each piece?
3. What must an admin be able to see and how?

---

## 2 The One Identifier: `session_id`

**`session_id` is the only public identity for a conversation.**

It is a caller-supplied or frontend-generated UUID that uniquely identifies one
multi-turn conversation between a user and an agent.

### 2.1 Rules

- `session_id` is the primary key used in every public API, every CLI command,
  every log line, and every metric dimension that refers to a conversation.
- `session_id` must never be called `thread_id`, `conversation_id`, or any
  other synonym in any public-facing surface (API, CLI, docs, UI).
- `session_id` is generated by the frontend (or CLI) before the first message
  is sent and remains stable for the lifetime of the conversation.
- For one-shot calls with no `session_id`, the runtime generates a per-request
  UUID. That UUID is ephemeral and not tracked by any registry. One-shot calls
  produce checkpoint state that will never be resumed.

### 2.2 The `thread_id` Implementation Note

Internally, LangGraph requires a key named `thread_id` in its `configurable`
dict to address checkpoint state. Fred maps `session_id → thread_id` at the
adapter boundary in `react_message_codec.py`:

```python
configurable["thread_id"] = config.session_id
```

This mapping is a **private implementation detail of the LangGraph adapter**.

It must never appear in:
- any public API response field
- any CLI command name or output label
- any documentation aimed at developers or operators
- any log line that is shown to end users

The LangGraph checkpoint tables (`checkpoints`, `blobs`, `writes`) store the
value under a column named `thread_id` because that is what LangGraph creates.
This column name is also an implementation detail. The Fred runtime queries it
internally and surfaces results exclusively under `session_id`.

## 3 Complete Conversation Record

A complete conversation record requires the following fields. All must be
available for admin queries, retention policies, and audit.

| Field | Source | Stored in | Required |
|---|---|---|---|
| `session_id` | Frontend / CLI | `session_history` (PK), checkpoint tables | ✅ always |
| `user_id` | Keycloak token / `ctx["user_id"]` | `session_history` (PK) | ✅ always |
| `team_id` | Execution context | `session_history` | ✅ managed exec |
| `agent_instance_id` | `RuntimeExecuteRequest` | `session_history` | ✅ managed exec |
| `created_at` | First message timestamp | derivable from `MIN(rank)` row | derived |
| `last_active_at` | Last message timestamp | derivable from `MAX(rank)` row | derived |
| `exchange_id` | Per-turn UUID | `session_history` | ✅ per message |

For no-security / dev mode: `user_id` defaults to `"unknown"` and `team_id` defaults
to `"personal"`. These must still be persisted so queries remain consistent.

## 4 Data Ownership Split

The two types of conversation data have distinct owners and must never be merged.

### 4.1 Session History (Message Content)

**Owner: `fred-runtime`**

Stored in the `session_history` table. Contains:
- every message in the conversation (user, assistant, tool calls, tool results)
- exchange grouping (`exchange_id`)
- timestamps, model metadata, token usage, sources
- `team_id` and `agent_instance_id` for filtering

Accessed via:
- `GET /agents/sessions/{session_id}/messages` — full message list for one session
- `GET /agents/sessions` — session list for one user (or all users for admin)

**Control-plane must not proxy or cache message content.** If the frontend needs
message history, it calls runtime directly using the `messages_url_template`
from `ExecutionPreparation`.

### 4.2 Session Metadata

**Owner: `control-plane-backend`** _(target state — implementation pending Phase 3b/5D)_

Will contain:
- session title (user-editable or auto-generated)
- creation timestamp
- last activity timestamp
- status (active, archived, deleted)
- preferences (language, display settings)
- `agent_instance_id` and `team_id` for sidebar grouping

Session metadata is created by control-plane at `prepare-execution` time or
on first turn. It is never stored in `fred-runtime`.

Until control-plane session metadata is implemented, the sidebar omits session
listing. The intentional placeholder (no session list in sidebar) is acceptable.
Adding a session list before the backend is ready is not.

### 4.3 Checkpoint State (Graph Resumability)

**Owner: `fred-runtime` checkpointer**

Stored in LangGraph tables (`checkpoints`, `blobs`, `writes`). Contains:
- serialized graph state enabling HITL resume and multi-turn continuity
- keyed internally by `session_id` (stored in LangGraph's `thread_id` column)

Accessed via:
- `GET /agents/checkpoints` — list all sessions with checkpoint state
- `GET /agents/checkpoints/{session_id}` — all checkpoints for one session
- `DELETE /agents/checkpoints/{session_id}` — purge checkpoint state for a session

Checkpoint state and message history are independent. Deleting checkpoints
does not delete message history. Deleting message history does not delete
checkpoints.

## 5 Admin Observability Requirements

The following capabilities are **mandatory** for any operator or system admin
managing a Fred deployment. They must be achievable from the CLI without the
frontend.

### 5.1 What an admin must be able to see

| Requirement | CLI command | API endpoint |
|---|---|---|
| List all sessions for a user | `/sessions [user_id]` | `GET /agents/sessions?user_id=<id>` |
| List all sessions across all users | `/sessions --all` | `GET /agents/sessions` (no filter, admin guard) |
| List all sessions for a team | _(pending)_ | `GET /agents/sessions?team_id=<id>` |
| List all sessions for a managed agent | _(pending)_ | `GET /agents/sessions?agent_instance_id=<id>` |
| Read conversation messages | `/history <session_id>` | `GET /agents/sessions/{session_id}/messages` |
| List checkpoint state | `/checkpoints` | `GET /agents/checkpoints` |
| Inspect checkpoints for one session | `/checkpoint <session_id>` | `GET /agents/checkpoints/{session_id}` |
| Purge checkpoint state for a session | _(CLI pending)_ | `DELETE /agents/checkpoints/{session_id}` |
| Pod storage stats | `/stats` | `GET /agents/checkpoints/_stats` |

### 5.2 Mandatory CLI behavior

The CLI (`fred-agent-chat`) must show, for any `/checkpoints` or `/sessions` listing:

- `session_id` — the 36-character UUID identifying the conversation
- `user_id` — who owns the conversation
- `team_id` — which team scope (when available)
- `agent_instance_id` — which managed agent (when available)
- `latest_created_at` — when the last turn happened
- a `◀` marker on the currently active session
- a `pending` warning when checkpoint writes are uncommitted (indicates a crashed turn)

For `/history <session_id>`, the CLI must show:
- each message with its `role`, `channel`, `timestamp`, and text content
- tool calls and results inline, clearly labeled
- model name and token usage in the final assistant message footer

### 5.3 What admin must never need to know

- LangGraph's internal `thread_id` column name
- Checkpoint blob structure or serialization format
- The physical path of the SQLite file or Postgres DSN
- Pod-internal service names or IP addresses

## 6 Retention And Purge Model

### 6.1 Principles

- **Retention policy is owned by control-plane**, not by `fred-runtime`.
- **Purge execution targets runtime APIs**, not direct DB access.
- `session_history` and checkpoint state are purged independently.
- A retention rule must be expressible as:
  `DELETE all sessions older than N days for team T`
  without knowing which pod or database they live on.

### 6.2 Purge surfaces (target state — not yet implemented)

| Target | Planned endpoint |
|---|---|
| Checkpoint state for one session | `DELETE /agents/checkpoints/{session_id}` ✅ exists |
| Message history for one session | `DELETE /agents/sessions/{session_id}` _(pending)_ |
| All data for one session | Combined call to both above _(pending)_ |
| Bulk purge by team / age | `POST /agents/sessions/purge` with policy filter _(pending)_ |

### 6.3 Purge queue note

The `session_purge_queue` table in `control-plane-backend` is a **legacy concept
inherited from `agentic-backend`**. It is not connected to the `session_history`
table written by `fred-runtime`. It must not be used as the retention mechanism
for runtime sessions. When a proper retention policy is implemented, it must
call the runtime purge endpoints above, not the legacy queue.

## 7 Session Lifecycle

```
Frontend / CLI
  │
  │  1. generate session_id (UUID)
  │  2. call prepare-execution → ExecutionPreparation
  │  3. POST /agents/execute/stream  { session_id, agent_instance_id, ... }
  │
Fred Runtime
  │
  │  4. persist turn to session_history  { session_id, user_id, team_id, agent_instance_id }
  │  5. persist checkpoint state         { thread_id=session_id (internal) }
  │  6. emit TurnPersistedEvent          { session_id }
  │
  │  ... subsequent turns reuse the same session_id ...
  │
Control-plane  (target state)
  │
  │  7. create session metadata record at prepare-execution or first turn
  │     { session_id, user_id, team_id, agent_instance_id, created_at, title }
```
